## TL;DR — Qtodo 长期愿景

**一句话**：让用户在任何设备（桌面 Android Windows、微信公众号）通过 AI 操作 Task，AI 在云端完成，桌面 app 同步数据并管理权限。

**三阶段独立可工作**：

| 阶段 | 范围 | 交付 |
|---|---|---|
| **阶段 1（已发 #7-#10）** | 本地持久化 | 冷启动不丢 Task |
| **阶段 2（即将拆 issue）** | 云端同步 | 桌面 + 云端双向同步 |
| **阶段 3（即将拆 issue）** | AI 操作 + 微信公众号 | 微信里跟 AI 聊，操作 Task |
| **阶段 4（即将拆 issue）** | 权限管理 | 桌面 app 设置 AI 权限 → 同步到云端 |

## 核心架构

```
┌────────────────────────────────────────────────────────────────────┐
│                            用户                                     │
└──────────┬─────────────────────────────────────┬──────────────────┘
           │                                     │
     桌面 Qtodo app                          手机（微信）
     (Tauri + 本地 SQLite)                   （微信公众号）
     [权限编辑器]                                │
           │ ↕ WebSocket                       │ ↕ 公众号消息
           │   (双向同步)                       │
           ▼                                     ▼
     ┌─────────────────────────────────────────────────────┐
     │    服务端（Cloudflare Workers + D1 SQLite）            │
     │  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
     │  │ 同步层    │  │ AI 代理层 │  │ 权限层    │           │
     │  │ (WS)     │  │ (LLM)    │  │ (查表)   │           │
     │  └────┬─────┘  └────┬─────┘  └────┬─────┘           │
     │       └────────┬────┴────────────┘                  │
     │                ▼                                     │
     │       ┌──────────────────┐                           │
     │       │ D1 (SQLite)      │                           │
     │       │ - tasks          │                           │
     │       │ - permissions    │                           │
     │       │ - sync_log       │                           │
     │       └──────────────────┘                           │
     └────────────────┬────────────────────────────────────┘
                      │
                      ▼
              ┌──────────────────┐
              │ 云 LLM API        │
              │ (DeepSeek/智谱等) │
              └──────────────────┘
```

**关键设计**：

1. **云端是数据权威源**（不是本地）
2. **AI 在云端操作 db**（24h 在线，不依赖桌面 app）
3. **桌面 app 同步云端**（WebSocket 长连接 + 离线时本地缓存 + 联网时回传）
4. **权限由桌面 app 设置**，**同步到云端**（云端是权限的 source of truth，桌面 app 是编辑器）
5. **AI 操作前查云端权限**（默认安全：删/归档要二次确认）

## 阶段 1 → 阶段 2 的过渡

阶段 1 完成时（#7-#10 跑通）：
- 本地 SQLite 是 source of truth
- 桌面 app 是 db 主人
- AI 操作接口预留但不实现（#P2 暴露的 `save_task` 等就是未来 AI 调用接口）

阶段 2 启动时（云同步）：
- **本地 SQLite 降级为"本地缓存"**
- 引入服务端 D1 作为 source of truth
- 桌面 app 启动时拉云端全量，运行时增量大
- 桌面 app 离线时建 Task 进本地队列，联网时上传
- 同步协议：WebSocket 长连接（桌面主动连服务端）+ 增量同步

**对阶段 1 的影响**（#7-#10 不动）：
- #P2 的 `save_task` 等 command **保留**——这些是阶段 3 AI 操作的接口
- #P2 的 schema **不变**——阶段 2 改的是"同步层"，不动业务表结构
- #P2 不需要预留"sync_version"等字段——**阶段 2 在 D1 里加，本地缓存不直接对应**

## 阶段 2：云端同步（拆 issue 中）

### 用户故事
> "我在公司电脑建了 Task，回家打开家里电脑，Task 在。"

### 核心设计
- **服务端**：Cloudflare Workers + D1
- **同步协议**：WebSocket 长连接（桌面主动连）
- **冲突解决**：last-write-wins（按 `updated_at` 字段）
- **离线策略**：本地 SQLite 作为离线缓存，联网时回传

### Schema 增量（D1 侧，不影响本地）
- `tasks` 表加：`updated_at`（已有）、`deleted_at`（软删）、`client_id`（创建它的设备）
- `sync_log` 表：记录所有变更（who/when/what）
- `devices` 表：注册过的桌面设备列表
- `permissions` 表：AI 权限设置（详见阶段 4）

### 桌面 app 改造
- 启动时 `connect_websocket()` → 服务端
- 收到服务端推送 → 拉变更 → 更新本地 SQLite
- 桌面 app 离线时建 Task → 写本地 + 入队 → 联网时上传
- 冲突时按 `updated_at` 取最新

### Issue 拆分（依赖顺序，**待 `to-issues` 技能发**）
1. **服务端基础设施**：Cloudflare Workers 项目骨架 + D1 schema + 部署
2. **桌面 app WebSocket 连接**：启动时连、断开重连、心跳
3. **桌面 → 云端 推送**：本地有变更时推服务端
4. **云端 → 桌面 拉取**：桌面订阅变更流
5. **冲突解决**：last-write-wins
6. **离线队列**：本地建 Task 排队、联网时回传

## 阶段 3：AI 操作 + 微信公众号（拆 issue 中）

### 用户故事
> "路上想加个 Task，微信跟 Qtodo 助理说'明天下午 3 点开会'，AI 自动建好，回到家打开桌面 app 看，Task 已经在。"

### 核心设计
- **公众号**：个人订阅号（不要企业资质）
- **AI 入口**：用户在公众号发消息 → 服务端 AI 解析 → 执行
- **AI 模型**：云 LLM（DeepSeek / 智谱 / 通义千问，OpenAI 兼容 API）
- **意图解析**：LLM 产"意图 + 参数"，服务端执行（**LLM 不能信**）

### 公众号架构
```
用户微信
  ↕ 客服消息
公众号（订阅号）
  ↕ 开发者服务器回调
服务端（Workers）
  ↕ 调云 LLM API
LLM 解析 → 返回 {action, params}
  ↕
服务端执行 → 写 D1 db
  ↕
桌面 app WebSocket 收到推送 → 同步
  ↕
用户看到："已建 Task：明天下午 3 点开会"
```

### 权限设计
- AI 操作前查 D1 的 `permissions` 表
- **默认行为**（防误操作）：
  - **建 Task** → AI 直接做
  - **改 Task**（改 due/priority/title）→ AI 直接做
  - **标 Complete** → AI 直接做
  - **删除** → AI 发客服消息"确认删除 X 吗？"，用户点确认按钮才执行
  - **归档** → 同删除，二次确认

### Issue 拆分（依赖阶段 2，**待 `to-issues` 技能发**）
1. **微信公众号开发者配置**：订阅号申请、服务器 URL 校验、token 验证
2. **客服消息接收 + 回复**：用户发消息 → 服务端接收 → 自动回复
3. **AI 意图解析层**：调云 LLM、解析自然语言、产 {action, params}
4. **AI 操作执行层**：根据 action 调 D1 db
5. **二次确认卡片**：删/归档时发客服消息带"确认/取消"按钮
6. **AI 操作日志**：所有 AI 操作记入 `ai_actions` 表（可审计）

## 阶段 4：权限管理（拆 issue 中）

### 用户故事
> "我在桌面 app 设置'不允许 AI 直接删除'，AI 在微信上发'删除 task-3'时，会先发确认卡片给我点。"

### 核心设计
- **桌面 app 是"权限编辑器"**：用户改权限 → 同步到 D1
- **D1 是"权限权威源"**：AI 操作前查 D1
- **权限粒度**：操作类型（建/改/标完成/删/归档）独立开关
- **权限是用户级**：未来多用户时每个用户独立（先单用户）

### 权限表 schema（D1 侧）
```sql
CREATE TABLE permissions (
  user_id TEXT NOT NULL,        -- 单用户时固定为 'default'
  operation TEXT NOT NULL,       -- 'create' | 'update' | 'complete' | 'delete' | 'archive'
  require_confirm INTEGER NOT NULL DEFAULT 0,  -- 1 = 需要二次确认
  enabled INTEGER NOT NULL DEFAULT 1,          -- 0 = 禁用
  updated_at TEXT NOT NULL
);
```

### 默认权限
| 操作 | enabled | require_confirm |
|---|---|---|
| create | 1 | 0 |
| update | 1 | 0 |
| complete | 1 | 0 |
| delete | 1 | **1** |
| archive | 1 | **1** |

### Issue 拆分（依赖阶段 3，**待 `to-issues` 技能发**）
1. **D1 加 permissions 表 + 迁移脚本**
2. **桌面 app 设置面板加"AI 权限"区块**：5 个开关（每个操作）
3. **桌面 → 云端 权限同步**：改设置后推服务端
4. **AI 操作前查权限 + 二次确认逻辑**
5. **权限变更日志**：记 `permission_changes` 表

## 关键决策汇总

| 决策 | 结论 | 理由 |
|---|---|---|
| 阶段 1 schema | 不预留云同步字段 | YAGNI，阶段 2 改 D1 侧 schema 不影响本地 |
| 数据权威源（阶段 2+） | **D1（云端）** | 支持 AI 24h 操作、不依赖桌面在线 |
| 同步协议 | **WebSocket 长连接** | 实时性 + 简单（vs 轮询 vs SSE） |
| 冲突解决 | **last-write-wins** | 简单、个人项目不需要 CRDT |
| 离线策略 | **本地 SQLite 缓存 + 联网时回传** | 用户体验优先、复杂度可控 |
| AI 模型 | **云 LLM（DeepSeek/智谱）** | 国内便宜、OpenAI 兼容 API |
| AI 跑在哪 | **服务端** | 24h 在线、不依赖桌面 app |
| 公众号类型 | **个人订阅号** | 不要企业资质、覆盖广 |
| 公众号能力 | **客服消息 + 富文本 + 按钮** | 个人订阅号最大能力 |
| 权限 source of truth | **D1（云端）** | AI 操作前必须能查到 |
| 权限编辑器 | **桌面 app** | 用户在桌面设，桌面同步给云端 |
| 默认权限策略 | **建/改/标完成直接做，删/归档二次确认** | 平衡体验和安全 |
| 软删/硬删 | **软删**（`deleted_at` 字段） | 同步时区分"删了"和"从未存在" |

## 范围明确不做（防止范围蔓延）

- ❌ **完整微信小程序**（要企业资质 + 审核）—— 公众号够用
- ❌ **vivo/小米/华为各厂商推送 SDK**（工作量 ×N）—— 公众号统一收口
- ❌ **Tauri Mobile / Flutter / RN** —— 公众号 + Web 覆盖
- ❌ **多用户 SaaS** —— 阶段 2/3/4 都是单用户（`user_id` 固定为 `'default'`），多用户留给"如果将来真有人用"再说
- ❌ **CRDT / Operational Transform** —— last-write-wins 够用
- ❌ **完整的 GTD 助手**（Inbox / Recurring / Subtask）—— 不是这个愿景的范围
